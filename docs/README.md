## 🎯 OctoAcme Core Principles

OctoAcme's project management is built on five foundational principles:

1. **Customer-first**: Prioritize customer value and usability in all decisions
2. **Iterative delivery**: Deliver small, testable increments to gather feedback early
3. **Clear ownership**: Each project has named leaders with explicit accountability
4. **Data-informed decisions**: Measure impact and iterate based on evidence
5. **Psychological safety**: Encourage feedback, learning, and blameless retrospectives

## 🔄 Communication Cadence

All OctoAcme projects follow this communication rhythm:

- **Daily**: 15-minute standups (focus: progress, blockers, dependencies)
- **Weekly**: 
  - PM + PdM alignment sync
  - Delivery team sync (show progress, update risks)
  - Status updates to stakeholders
- **Monthly**: Executive stakeholder briefings
- **Per milestone/sprint**: Demo/Review sessions
- **As needed**: Ad-hoc escalations and incident communications
- **Post-project**: Retrospective (45–75 minutes)

## 📊 Key Artifacts

Every OctoAcme project maintains these core artifacts:

1. **Project Charter / One-pager** — Problem, goal, success metrics, stakeholders, timeline
2. **Roadmap & Release Plan** — Phased delivery with milestones
3. **Sprint/Iteration Backlog** — Prioritized, estimated, ready-to-build work
4. **Acceptance Criteria & Definition of Done** — Quality standards
5. **Risk Register** — Tracked, monitored, and escalated risks
6. **Retrospective Notes & Action Items** — Learnings and improvements

## ❓ FAQ

**Q: Where do I start if I'm new to OctoAcme?**  
A: Read the [Project Management Overview](octoacme-project-management-overview.md) first, then your role-specific docs from the navigation section above.

**Q: How do I propose a new process or update an existing one?**  
A: Open an issue using the **[Process Doc Update]** template in `.github/ISSUE_TEMPLATE/`.

**Q: What if I disagree with a process?**  
A: Raise it in your retrospective, then escalate via a structured issue for discussion. OctoAcme thrives on feedback.

**Q: Where do I store project-specific guidance?**  
A: Keep it in your project repo's `.copilot/` directory so Copilot Spaces can use it as context.
